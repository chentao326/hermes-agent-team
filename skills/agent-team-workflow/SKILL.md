# 多智能体协作工作流

## 概述

模拟一个完整的技术团队，按角色分工协作处理需求。支持两种模式：**分析模式**（全角色评审）和**执行模式**（开发 + 审查）。

每个角色有独立的专业视角，最终输出结构化的综合方案。

## 触发条件

用户提到以下关键词时加载此 skill：
- "团队协作"、"多角色"、"智能体协作"
- "全流程"、"上下游一起看"
- "让 XXX 角色看看"
- "按团队流程走"

## 角色定义

每个角色在分析时遵循以下思维模式：

**CEO**: 商业价值、市场时机、ROI、风险评估、资源优先级
**PM**: 用户痛点、需求真伪、功能优先级、用户故事、验收标准
**架构师**: 系统设计、技术选型、扩展性、安全性、trade-off 分析
**后端**: API 设计、数据模型、性能、工程实践、具体实现
**前端**: UI/UX、组件设计、交互细节、性能优化、具体实现
**QA**: 测试策略、用例设计、边界条件、自动化方案
**Reviewer**: 代码质量、安全审查、最佳实践、改进建议

---

## 模式一：分析模式（默认）

适用于：需求评审、技术方案讨论、架构设计。

### 流程

1. 用户提出需求/问题
2. 按需选择角色顺序（默认：CEO → PM → 架构师 → 后端 → 前端 → QA → Reviewer）
3. 每个角色以该角色身份输出分析，格式如下：

```
## [角色名] 视角

[从该角色角度分析，包含核心判断、关键问题、具体建议]
```

4. 所有角色分析完成后，输出「综合结论」：

```
## 综合结论

**核心决策**: [一句话总结]
**优先级**: [P0/P1/P2 排序]
**风险点**: [主要风险 + 应对]
**下一步**: [具体行动项]
```

### 快捷指令

用户可以指定只走部分角色：
- "让 PM 和架构师看下" → 只走 PM + 架构师
- "CEO 视角" → 只用 CEO 视角
- "全流程评审" → 走全部 7 个角色

### 角色顺序可调

默认顺序适用于从 0 到 1 的新项目。已有项目可以跳过 CEO/PM，直接从架构师开始。

## 模式二：执行模式

适用于：已有明确需求，需要编码实现。

### 流程

1. **PM** 确认需求和验收标准
2. **架构师** 输出技术方案（若已有方案则跳过）
3. **后端 + 前端** 并行派发 delegate_task 执行开发
4. **QA** 设计并运行测试
5. **Reviewer** 审查代码
6. **CEO** 最终验收

执行模式使用 delegate_task 派发实际编码任务，子代理的 context 中注入对应角色的思维模式。

### delegate_task 角色模板

派发子代理时，在 context 开头注入角色指令：

```
你是一名[角色名]：[角色核心能力一句话]。

执行以下任务时遵循该角色的专业标准和关注点：
- [3-5 个具体关注点]

任务：
[具体任务描述]
```

---

## 角色切换

在分析模式中，不要实际调用 `/personality` 切换（会改变系统 prompt），而是通过改变输出的思维框架来模拟角色。

如果用户确实需要切换当前 agent 的 persona，提醒使用 `/personality <角色名>` 命令。

---

## 注意事项

1. **分析模式不写代码** — 只输出分析、方案、建议。用户确认后再进入执行模式
2. **角色输出要体现差异** — CEO 关注商业，PM 关注用户，架构师关注系统，开发关注实现
3. **冲突要标注** — 如果不同角色意见不一致，在综合结论中标注 trade-off
4. **时间意识** — 评估每个方案的工期和资源需求
5. **中文输出** — 所有角色输出使用中文
